# mipt_contests

Репозиторий с материалами по контестам и практическим заданиям за несколько семестров:

- **C++ — 1 семестр**: 14 контестов
- **C++ — 2 семестр**: 11 контестов
- **Python — 3 семестр**: 3 контеста и финальный контест
- **ИТ-технологии — 4 семестр**: набор отдельных практических задач и мини-проектов

Основная идея репозитория: в каждом разделе собраны **условия задач**, **сохранённые решения**, а для части разделов — ещё и **служебные файлы для сборки, тестов и запуска**.

---

## Структура репозитория

```text
.
├── C++_sem_1/
│   └── contest_*/
│       ├── readme.md
│       └── solutions/
├── C++_sem_2/
│   └── contest_*/
│       ├── readme.md
│       └── solutions/
├── python_sem_3/
│   └── contest_*/
│       ├── readme.md
│       └── solutions/
└── IT_tech_4_sem/
    ├── task0/
    ├── task1/
    ├── task-cmake/
    ├── task-docker/
    └── task-testing-cpp/
```

### Что находится внутри контестов

Обычно каждый каталог `contest_*` содержит:

- `readme.md` — оформленные условия задач;
- `solutions/` — сохранённые решения (`.cpp`, `.py`, а в отдельных случаях дополнительные файлы).

Во многих `readme.md` также есть краткая сводка по задачам:

- статус решения;
- видимые вердикты (`OK`, `WA`, `RE` и т. д.);
- информация о том, сохранилась ли посылка на странице контеста.

## Разделы по семестрам

### C++_sem_1

Первый семестр по C++.

- **14 контестов**
- **83 файла с решениями**
- в большинстве контестов структура приведена к единому виду: `readme.md` + `solutions/*`

Особый случай:

- `C++_sem_1/contest_11/solutions/A_cstring/` — задача с несколькими файлами (`cstring.h` и `cstring.cpp`), а не одним исходником.

### C++_sem_2

Второй семестр по C++.

- **11 контестов**
- **31 файл с решениями**
- часть контестов содержит неполные наборы решений: это нормально для репозитория, так как сохранённый код зависит от того, была ли доступна страница посылки

### python_sem_3

Третий семестр по Python.

- **4 каталога с контестами**: `contest_1`, `contest_2`, `contest_3`, `contest_final`
- **19 файлов с решениями**
- решения сохранены в формате `.py`

### IT_tech_4_sem

Практические задания по ИТ-технологиям за 4 семестр.

#### task0

Минимальный shell-скрипт `run.sh`, выводящий `Hello World`.

#### task1

Shell-скрипт для резервного копирования файлов по расширению:

- рекурсивно обходит директорию;
- копирует файлы с нужным расширением в папку бэкапа;
- создаёт `tar.gz`-архив.

#### task-cmake

Мини-проект на C++ с CMake:

- генерация заголовочного файла через Python;
- библиотека с функцией сортировки;
- исполняемый файл с примером работы;
- тесты на GoogleTest.

#### task-docker

Небольшой проект с:

- CMake-сборкой C++-части;
- Flask-приложением `app.py`;
- Dockerfile для запуска внутри контейнера.

Маршрут Flask-приложения:

- `GET /app` → `ok`

#### task-testing-cpp

Самый объёмный практический проект в репозитории.

Содержит:

- CMake-проект на C++17;
- unit-тесты на GoogleTest / GMock;
- примеры;
- работу с деревом файлов;
- простую библиотеку функций;
- weather-модуль с HTTP-запросами через `cpr`;
- генерацию отчёта покрытия.

Для сборки используются внешние зависимости, которые подтягиваются через `FetchContent`.

## Как пользоваться репозиторием

### Быстрый просмотр решений

Открывайте нужный семестр и затем конкретный контест:

```text
C++_sem_1/contest_1/readme.md
C++_sem_1/contest_1/solutions/
```

или

```text
python_sem_3/contest_2/readme.md
python_sem_3/contest_2/solutions/
```

### Сборка проектов из IT_tech_4_sem

#### task-cmake

```bash
cd IT_tech_4_sem/task-cmake
mkdir -p build
cd build
cmake ..
make
```

#### task-testing-cpp

```bash
cd IT_tech_4_sem/task-testing-cpp
./build.sh
```

#### task-docker

Пример запуска через Docker:

```bash
cd IT_tech_4_sem/task-docker/cmake-with-flask
docker build -t cmake-with-flask .
docker run --rm -p 8898:8898 cmake-with-flask
```

После запуска приложение должно отвечать на `http://localhost:8898/app`.

## Важные замечания

### 1. В репозитории не найдено явных секретов

Явных ключей, токенов, паролей и приватных credential-файлов в содержимом репозитория нет.

Что нашлось:

- упоминание файла `api_key.txt` в `IT_tech_4_sem/task-testing-cpp`;
- строка-заглушка `"Your API Key"` в примере weather-модуля.

То есть механизм работы с API есть, но **сам ключ в репозиторий не положен**.

### 2. В корне нет лицензии

Файла `LICENSE` в репозитории нет. Если репозиторий планируется публиковать или показывать как портфолио, лучше добавить лицензию отдельно.

### 3. Часть задач и проектов ориентирована на Linux / WSL / Unix-подобную среду

Это видно по:

- наличию `run.sh` и `build.sh`;
- использованию `bash`;
- CMake-сборке;
- Dockerfile;
- зависимостям, которые удобнее ставить в Linux-среде.

На Windows без WSL часть команд может потребовать адаптации.

### 4. Для `task-testing-cpp` нужен интернет во время первой сборки

Проект подтягивает внешние зависимости (`googletest`, `cpr`) через `FetchContent`, поэтому для первой полной сборки нужен доступ к сети.

### 5. В `task-testing-cpp` есть привязка к приватному GitLab в README

Внутренний `README.md` этого проекта содержит команды с приватным `git clone` из GitLab. Это не проблема для самого репозитория, потому что исходники уже лежат здесь, но для внешнего читателя такие команды могут быть бесполезны.

### 6. В `task-cmake` есть неточность в описании сложности

Внутренний README проекта упоминает сортировку выбором «за O(n)». Для классической сортировки выбором это некорректно: стандартная асимптотика — **O(n²)**.

### 7. Некоторые контесты могут быть неполными

Это ожидаемо: не для каждой задачи обязательно сохранилась страница с кодом посылки. Поэтому:

- условие задачи может быть в `readme.md`;
- а файла решения в `solutions/` может не быть.
