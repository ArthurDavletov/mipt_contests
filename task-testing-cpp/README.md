## Установка и запуск

Вы можете с лёгкостью установить данный проект и протестировать у себя, запустив следующие команды:

```sh
git checkout task-testing-cpp
cd task-testing-cpp

```

## Что представляют собой тесты?

### 01-simple-library

#### AddTestCase

Комплект тестов `AddTestCase` тестирует функцию `Add(int x, int y)` на диапазоне `-10 <= x <= 10` и `-10 <= y <= 10`. В результате этого тестируется сложение чисел с разными, одинаковыми знаками, а также целых и нулей.

#### LeapTestCase

Комплект тестов `LeapTestCase` более комплексный. Согласно [Википедии](https://ru.wikipedia.org/wiki/Високосный_год):

* год, номер которого кратен 400, — високосный;
* остальные годы, номер которых кратен 100, — невисокосные;
* остальные годы, номер которых кратен 4, — високосные;
* все остальные годы — невисокосные.

Отчего комплект тестов можно разделить на следующие тесты:

* `NonLeapYears`. Проверка годов, которые не являются високосными. В большинстве случаев сначала находят именно високосные года.
* `SpecialNonLeapYears`. Проверка годов, которые кратны 100, но не кратны 400
* `LeapYears`. Проверка обычных високосных годов, кратных 4.
* `SpecialLeapYears`. Проверка годов, которые кратны 400.
* `InvalidYear`. Проверка отрицательных годов. Отработка исключения `std::invalid_argument`

В каждом тесте 9-15 годов. Суммарно проверка на 57 различных годах.

Для проверки `GetMonthDays` есть следующие тесты:

* `LeapFebruary`. Проверка февралей високосных годов.
* `NonLeapFebruary`. Проверка февралей невисокосных годов.
* `Days30`. Проверка всех 30-дневных месяцев
* `Days31`. Проверка всех 31-дневных месяцев
* `USSRCalendar`. Проверка того, что в каждом месяце в 1930 году по 30 дней
* `InvalidMonth`. Проверка месяцев, не входящих в отрезок [1, 12].

Суммарно 159 тестов.

### 02-tree

Данная библиотека является аналогом утилиты `tree`. `GetTree(std::string path)` возвращает `FileNode`, где хранятся вектор из `FileNode` (дочерних файлов), булевой переменная `is_dir` и название самого файла или директории. Библиотека использует `<filesystem>`

#### TreeTestCase

Для удобства тестирования были введены `void RecursiveCheck(FileNode, fs::path, bool)` и `bool AnyEmptyDirectories(fs::path)`.

`RecursiveCheck` проверяет, является ли `FileNode` директорией. Если она является таковой, то проверяется корректность размера вектора `chidren`. А далее функция рекурсивно вызывается для дочерних директорий.

`AnyEmptyDirectories` проверяет наличие хотя бы одной пустой директории

Перед тестированием (`SetUpTestSuite`) создаётся следующая структура:

```sh
test
├── 1
│   ├── 100
│   └── test.py
├── 2
│   ├── 200
│   └── file.cpp
└── test.txt

4 directories, 3 files
```

После проведения всех тестов (`TearDownTestSuite`) директория `test/` удаляется

Структура комплекта тестов `TreeTestCase` следующая:

* `PathNotExists`. Проверка несуществующей директории. Обработка исключения `std::invalid_argument`.
* `InvalidPath`. Попытка запустить GetTree от файла. Обработка исключения `std::invalid_argument`.
* `DirsOnlyTrue` и `DirsOnlyFalse`. Проверка команды GetTree при соответствующих значениях `dirs_only`.
* `FilterEmptyDirs`. Запуск функции очистки директории от пустых папок. Корректность проверяется функцией `RecursiveCheck`
* `RemoveCurrentFolder`. Создание пустой директории и попытка запустить очистку пустых директорий. Обработка исклчючения `std::runtime_error`
* `FileNodeEqCheck`. Проверка равенства двух одинаковых FileNode.

### 03-weather

Для выполнения задания были созданы класс `WeatherFake`, функции `FromFile` и `ToFile` и мок-класс `WeatherMock`. У классов были объявлены конструкторы и деструкторы (если их не было в явном виде, то были созданы по-умолчанию), так как по документации их создание занимает большое время при сборке проекта.

#### `class WeatherFake`

* Для сохранения результатов будет использовать структура `map<string, <map<string, crp::Response>> requests_cache_`. Допустим, мы обращаемся по адресу `http://ya.ru/` с параметром `{city: "Ufa"}`. Тогда ответ сервера будет записан в `requests_cache_["http://ya.ru/"]["Ufa"]`.
* Для того, чтобы результат сохранялся в систему, используем сохранение и загрузка из файлов с помощью функций `FromFile` и `ToFile`.

#### `map<string, map<string, Response>> FromFile(ifstream)`

Функция `FromFile` считывает данные с сохранённего json-файла.

#### `void ToFile(map<string, map<string, Response>>, ifstream)`

Функция записывает данные в файл.

#### WeatherMock

Мок-класс содержит мок-методы следующих функций:

* `GetTemperature`
* `GetTomorrowTemperature`
* `GetLocationKey`
* `Get`

#### WeatherTestCase

Перед запуском тестов программа пытается найти ключ API в директории сборки `api_key.txt`. Если файла нет, то ключ API будет пустым.

Данный комплект тестов состоит из следующих тестов:

* `GetDiffs`. Подделывается возврат функции `GetTemperature`, отчего тестируется результат функции `FindDiffBetweenTwoCities`. 25 тестов
* `GetLocationKeyCheck`. Проверка города, которого не существует. Обработка исключения `std::runtime_error`.
* `GetTempCheck`. Проверка температуры воздуха в Уфе в пределах от -110 до 70.
* `GetDiffString`. Подделывается вывод функции `GetTemperature`, за счёт которого проверяется функция получения строки с разницей температур двух городов.
* `BadApiError`. Составляется искусственный `crp::Response` с статус-кодом 404. Далее подделывается функция `Get` и обрабатывается исключение при плохом API.
* `TomorrowTempCheck`. Также, как и в `GetTempCheck`, но проверяется функция получения завтрашней температуры.
* `GetTomorrowDiffCheck`. Проверяется функция получения разницы температур сегодня и завтра. Для этого подделывается результаты функций получения сегодняшней (`GetTemperature`) и завтрашней (`GetTomorrowTemperature`) температур.
